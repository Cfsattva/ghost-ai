## Session Notes

- Generated components live under `components/ui/` — do not edit per `context/ai-workflow-rules.md`. Compose/restyle in app-level components instead.
- Base UI's `Input`/`Textarea` add a client-only `style={{caretColor: "transparent"}}`, which triggers a cosmetic React hydration warning in dev. It's internal to the protected `components/ui/*` files and does not affect rendering; not fixed here since it would require editing generated files.
- 02-editor-chrome judgment calls (spec was silent on these): `ProjectSidebar` takes an `onClose` callback alongside `isOpen` since the spec requires a working close button; `EditorNavbar` takes `isSidebarOpen`/`onToggleSidebar` rather than owning its own state, since actual sidebar-open state will belong to whatever page/layout composes these two chrome pieces together (not built yet — this feature unit only covers the standalone components, no editor route/page was created). The sidebar is a full-height (`inset-y-0`) fixed overlay with a higher z-index than the navbar, so it floats above the navbar too, not just the canvas, when open. The dialog pattern lives at `components/editor/editor-dialog.tsx` (colocated with the other chrome components since the spec didn't give it a path) and only adds the project's `rounded-3xl`/token styling on top of `components/ui/dialog.tsx` — no dialog is instantiated anywhere yet.
- 03-auth judgment call: the spec required `/` to redirect authenticated users to `/editor` and the editor navbar to carry `UserButton`, but no `/editor` route existed yet (02-editor-chrome only produced standalone components) and no feature spec for the editor page/canvas exists yet either. Rather than leaving `/editor` as a 404 redirect target, added a minimal `app/editor/page.tsx` that just assembles the existing `EditorNavbar` + `ProjectSidebar` with local sidebar-toggle state — no canvas, no data. This is intentionally the smallest thing that makes the auth redirect and `UserButton` real/verifiable; the actual editor workspace is still unbuilt and needs its own feature spec.
- Clerk's `Appearance` type takes `theme` (not `baseTheme`) for a prebuilt theme object like `dark` from `@clerk/ui/themes` — `appearance={{ theme: dark, variables: {...} }}`. Confirmed from the type doc comment in `@clerk/ui/dist/internal/appearance.d.ts`.
- 04-project-dialogs judgment calls (spec was silent on these): the hook holds the mock `Project[]` list itself, not just dialog/form/loading state, since something has to own the data the dialogs mutate and the spec's "Use mock project data only" strongly implies one place owns it — kept it in `hooks/use-project-dialogs.ts` rather than splitting into a separate mock-data module since the list is only 3 seed rows. `Project.isOwner` is the boolean driving both the sidebar's owned/shared tab split and whether the actions dropdown renders (spec: "Show actions only for owned projects. Hide actions for shared/collaborator projects."). The mobile backdrop scrim is scoped with `lg:hidden` so it only exists (and only intercepts outside-clicks) below the `lg` breakpoint, matching the spec's "On mobile" qualifier — desktop clicking outside the floating sidebar overlay intentionally does not close it, since the spec never asked for that on desktop. Base UI's `Dialog.Popup` already moves focus to the first tabbable element by default, which is what satisfies "input auto-focuses" for the Rename dialog — no explicit focus-management code was needed beyond ordering the input before the dialog's close button in the JSX tree. `Enter submits` is implemented by wrapping each of Create/Rename in a native `<form onSubmit>` with the action button as `type="submit"`, which is standard single-input-form behavior rather than a manual keydown handler.
